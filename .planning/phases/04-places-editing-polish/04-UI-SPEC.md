## Phase 4 Component Inventory

### 1. PlaceGroupHeader (new — within PhotoGallery)

Purpose: visually separate the photo grid into labeled groups when place/city names are available.

When to render:
- At least one photo in the country has a non-null `placeName` field (returned from server after geocoding)
- Photos without a place name are collected into an "Other photos" group rendered last

Visual specification:
- Full-width row spanning all 3 grid columns (rendered via CSS `grid-column: 1 / -1` or as a break in the grid)
- Height: 32px, vertical padding: 8px each side
- Background: `--color-accent-subtle` (rgba(59,130,246,0.15))
- Left edge: 3px solid `--color-accent` (#3b82f6) — left border accent stripe
- Text: `text-label` (14px), weight 600, `--color-text` (#111827)
- Format: "[City / Place Name]" — just the place name, no country repetition
- Right side: small pill badge — `text-label` (12px), weight 600, `--color-accent` text, no background, no border — showing photo count "N"
- No expand/collapse — groups are always open in Phase 4 (collapsible is a v2 enhancement)
- "Other photos" group header: same visual treatment, label is "Other photos"

Layout position:
- Inserted as a full-span row in the existing `grid grid-cols-3 gap-1 p-2` grid inside PhotoGallery
- Each group header precedes its photos in document order

Implementation note: PhotoGallery receives `photos` already sorted/grouped by the caller or internally groups them — this is an implementation decision for the planner, not a visual contract change.

### 2. GeocodingPlaceholder (new — in PinPopup + photo detail)

Purpose: show a "pending" state while a photo's place name is being resolved by the async Nominatim worker.

States:
- `pending` — place name not yet resolved (photo has GPS but `placeName` is null or absent)
- `resolved` — place name is present (`placeName` is a non-empty string)
- `unavailable` — geocoding exhausted / no result (server sets `placeName` to `""` or a sentinel)

Visual specification:
- `pending`: inline text "Geocoding location…" — `text-label` (14px), weight 400, `--color-text-muted`, no spinner (keep pin popup lightweight)
- `resolved`: display `placeName` string — `text-label` (14px), weight 400, `--color-text-muted`, beneath the photo filename in PinPopup caption area
- `unavailable`: display nothing — omit the caption line entirely (do not show "—" or empty string)

PinPopup caption area (extended from Phase 3):
- Line 1: filename truncated at 24 chars — `text-label` (14px), weight 400, `--color-text`
- Line 2 (conditional): place name OR "Geocoding location…" — `text-label` (14px), weight 400, `--color-text-muted`
- Padding: 8px horizontal, 8px vertical (same as Phase 3)

### 3. UploadProgressList (new — replaces statusMessage in PhotoUploadForm and GlobalUploadButton)

Purpose: show a per-file row with filename, progress bar, and final status for each file in the current upload batch.

Visual specification (container):
- Appears below the upload zone in `PhotoUploadForm`, replacing the single-line `statusMessage` `<p>` for batches
- For `GlobalUploadButton`: expands inside the existing drop-down card (`.w-72 bg-surface border border-border rounded-md shadow-sm p-3`)
- Container: `flex flex-col gap-1`, no extra padding beyond existing card wrapper

Visual specification (per-file row):
- Height: minimum 40px, `flex items-center gap-2`, `px-2`
- Filename: `text-label` (14px), weight 400, `--color-text`, truncated at 28 chars with "…", `flex-1 truncate`
- Progress bar: height 4px, width 80px, `--color-border` track, `--color-accent` fill, `border-radius: 2px`
  - Implemented as a `<div>` wrapper + inner fill div (no native `<progress>` — avoids cross-browser style reset complexity)
  - Fill width driven by percentage (0–100) passed as inline style `width: {pct}%`
  - While in-flight: bar animates from 0% to 90% (fake progress — true XHR progress not available with fetch); jumps to 100% on server 200
- Status indicator (replaces progress bar after upload completes):
  - Success: text "Done" — `text-label` (14px), `--color-text-muted`
  - Error: text "Failed" — `text-label` (14px), `--color-destructive`
- Max rows visible: 5; if batch > 5 files, container is `max-h-[200px] overflow-y-auto`

Auto-clear behavior:
- After all files in the batch are Done or Failed, wait 4 seconds, then clear the list (matching existing success auto-clear in Phase 1–3)
- Failed rows stay until the user dismisses or re-uploads (same pattern as existing error messages)

### 4. PhotoEditModal (new — overlay panel)

Purpose: allow the user to move a photo to a different country or reposition its pin on the map.

Trigger surface:
- PinPopup: a text button "Edit location" rendered below the place name caption area
- Button: `text-label` (14px), weight 600, `--color-accent` (#3b82f6) text, no border, no background, `min-h-11` touch target — same style as the "Select" toggle in PhotoGallery
- This is the ONLY trigger surface in Phase 4. The lightbox toolbar does not get a move/edit button (keep lightbox uncluttered).

Modal structure:
- Backdrop: `fixed inset-0 bg-overlay z-[600]` — above the sidebar (z-500) and upload button (z-500)
- Panel: `fixed inset-x-4 inset-y-4 md:inset-x-auto md:left-1/2 md:-translate-x-1/2 md:w-[420px] md:top-1/2 md:-translate-y-1/2 md:h-auto bg-surface rounded-lg shadow-xl flex flex-col z-[601]`
- Mobile: full-screen inset 16px all sides — inset-x-4 inset-y-4
- Desktop: centered 420px wide card, vertically centered

Modal header:
- `flex items-center justify-between px-6 py-4 border-b border-border flex-shrink-0`
- Heading: "Edit photo location" — `text-heading` (20px), weight 600, `--color-text`
- Close button: "Cancel" text — `text-label` (14px), `--color-text-muted`, `min-h-11 min-w-11`, top-right, same style as existing sidebar close button

Modal body (two sections):

**Section A — Move to a different country:**
- Label: "Move to a different country" — `text-label` (14px), weight 600, `--color-text`, `mb-1`
- `<select>` dropdown: same styling as existing GlobalUploadButton country select — `w-full min-h-11 px-2 bg-surface text-body text-text border border-border rounded-md outline-none focus:ring-2 focus:ring-accent`
- Placeholder option: "Keep current country ([CountryName])" — selected by default
- Changing the select immediately enables the "Save" button; no live preview

**Section B — Reposition pin (GPS-only):**
- Shown only when the photo has GPS coordinates (`lat` + `lng` present)
- Label: "Reposition pin" — `text-label` (14px), weight 600, `--color-text`, `mb-1`
- A mini `CountryPinMap`-style `MapContainer` (react-leaflet) — height 200px, full modal width, showing the current pin at `[lat, lng]`
- The pin in this mini map is draggable (`<Marker draggable={true}>`) — drag to reposition
- While dragging: pin shows a drag cursor; no other visual change
- After drag ends: `dragend` event updates local state with new `[lat, lng]`; the mini map does NOT auto-reassign country — the country select in Section A handles country assignment
- If photo has no GPS: Section B is omitted entirely — modal shows Section A only

Modal footer:
- `flex items-center justify-end gap-2 px-6 py-4 border-t border-border flex-shrink-0`
- "Cancel" button: `min-h-11 px-4 rounded-md text-label font-semibold text-text-muted hover:text-text transition-colors focus:outline-none focus:ring-2 focus:ring-accent`
- "Save" button: `min-h-11 px-4 rounded-md text-label font-semibold text-white bg-accent hover:bg-accent-dark transition-colors focus:outline-none focus:ring-2 focus:ring-accent` — disabled (opacity-50, cursor-not-allowed) when neither section has a change; enabled when country changed OR pin dragged
- While saving: "Save" button shows "Saving…", disabled; modal stays open until server 200; on success modal closes

Error state in modal:
- Inline error below footer buttons: `text-label` (14px), `--color-destructive`, role="alert" — "Save failed. Check your connection and try again."
- Modal stays open on error

### 5. Mobile Polish Targets (POL-06 — not a new component, targeted fixes)

These are specific layout and interaction contracts for the existing component set on mobile (viewport < 768px / below `md` breakpoint).

**CountrySidebar (mobile bottom panel):**
- Current implementation: `fixed bottom-0 left-0 right-0 h-[55dvh] border-t` — confirmed in source
- Phase 4 addition: drag handle strip at top of panel — `w-10 h-1 rounded-full bg-border mx-auto mt-2 mb-0 flex-shrink-0`
- No functional drag-to-resize in Phase 4 (visual affordance only — full drag mechanic is v2)
- `overflow-y: auto` on the inner scroll region must include `-webkit-overflow-scrolling: touch` — add `[&]:touch-action-manipulation` or inline style if Tailwind doesn't cover it

**GlobalUploadButton (mobile):**
- Current position: `fixed top-2 right-2 z-[500] w-72` — on mobile this extends 288px leftward from right edge, potentially overlapping map controls
- Phase 4 fix: on mobile (`< md`), constrain width to `w-[calc(100vw-16px)]` and anchor `right-2 left-2` when the dropdown card is open; the trigger button itself stays `right-2` at its natural width
- The trigger label button stays as-is (top-right, small) — only the expanded dropdown card gets full-width treatment

**CountryPinMap (mobile):**
- Phase 3 spec: 240px desktop, 180px mobile — already coded, confirmed in phase context
- Phase 4: no change to height; ensure `touch-action: pan-y` does not block map pan gestures — map container must use `touchAction: 'none'` or rely on Leaflet's default pointer event handling

**PhotoGallery (mobile):**
- 3-col grid is retained on mobile (thumbnails are ~29% viewport width — minimum ~87px on 320px screen — acceptable for touch selection)
- PlaceGroupHeader full-span rows must be tested at 320px — the place name text must truncate with overflow-hidden and not break the grid
- Select mode toolbar: minimum touch target for Delete and Cancel buttons is already coded at `min-h-11` — no change needed

**PinPopup (mobile):**
- Leaflet popup is 180px wide — fits within 320px screens; no change
- "Edit location" button: `min-h-11` touch target — contract enforces this (see §4 above)

**PhotoEditModal (mobile):**
- Full-screen inset-4 layout covers the entire screen on mobile — repositioned map (Section B) is 200px tall, leaving ~[dvh - 200px - header - footer] for Section A content
- On small screens (<480px) where modal height may be tight: Section B mini-map shrinks to 160px height

## Interaction Flows

### Flow 1: Place names appear after async geocoding (POL-01, POL-02)

1. User uploads a GPS photo — server stores it without `placeName`
2. Async worker picks up the queued photo (within seconds to minutes depending on queue depth)
3. Worker calls Nominatim (≤1 req/s), receives city/place name, patches the photo document
4. Client polls or refetches `usePhotos(countryCode)` — React Query's default `staleTime: 0` will re-fetch on window focus
5. **PinPopup**: if photo is open in a popup, next open shows resolved place name (popup does not live-update while open — acceptable for Phase 4)
6. **PhotoGallery**: on next `usePhotos` refetch, gallery re-renders with `PlaceGroupHeader` sections if ≥1 photo has a place name
7. **"Geocoding location…" state**: if user opens PinPopup for a photo whose `placeName` is null/absent, the placeholder text is shown; no spinner

### Flow 2: Group photos by place within a country (POL-02)

1. User opens a country — sidebar renders
2. `usePhotos` returns photos; component groups them client-side by `placeName`
3. Photos with the same `placeName` are collected under one `PlaceGroupHeader`
4. Photos with no `placeName` (null or empty) are grouped last under "Other photos" header — only if other groups exist; if ALL photos lack a place name, no group headers are shown (gallery renders as a flat grid, same as Phase 3)
5. No filter UI in Phase 4 — grouping is always on when data is available

### Flow 3: Edit photo location — move to different country (POL-03, country change)

1. User clicks a pin on the CountryPinMap → PinPopup opens
2. User clicks "Edit location" button in PinPopup → PinPopup closes, PhotoEditModal opens
3. Modal shows Section A (country select) and — if photo has GPS — Section B (mini map with current pin)
4. User picks a new country in the select — "Save" button becomes enabled
5. User clicks "Save" → loading state
6. Server PATCH updates `countryCode` (and optionally `lat`/`lng` if pin was also dragged)
7. React Query invalidates both the old and new country's photo lists
8. Modal closes; CountryPinMap and PhotoGallery for the old country no longer show this photo
9. If user deselects country and re-opens the new country: photo appears there

### Flow 4: Edit photo location — reposition pin (POL-03, pin reposition)

1. User clicks "Edit location" in PinPopup → PhotoEditModal opens
2. User drags the pin in Section B mini-map to the new position
3. Drag cursor active while dragging; new coordinates captured on dragend
4. Country select in Section A remains unchanged (user is just moving pin within same country)
5. "Save" button enabled
6. User clicks "Save" → server PATCH updates `lat`, `lng` (same `countryCode`)
7. React Query invalidates the country's photos; CountryPinMap re-renders with pin at new position

### Flow 5: Per-file upload progress (POL-05)

1. User picks or drops files in GlobalUploadButton or PhotoUploadForm
2. `UploadProgressList` renders immediately with one row per file (progress bar at 0% or animated indeterminate)
3. Files are sent — progress bar animates from 0% to 90% pseudo-progressively (animation-duration: 2s ease-out)
4. On server 200 for the batch: all bars jump to 100%; status text changes to "Done" for each succeeded file
5. Failed files (per-file error in `results[].error`): bar disappears, status text shows "Failed" in destructive color
6. After 4 seconds (all Done), list auto-clears
7. Failed rows persist with "Failed" label until user dismisses (clears whole list) or uploads again

## Accessibility Contract

Inherited from Phases 1–3 plus Phase 4 additions:

- `PlaceGroupHeader`: `role="rowgroup"` is not correct for a grid insert — use `role="separator"` with `aria-label="[Place Name] — N photos"` on the header row element
- `GeocodingPlaceholder` pending text: `role="status"` on the container `<p>` — announces when text changes from pending to resolved
- `UploadProgressList` container: `role="status"` — announces batch completion to screen readers
- Per-file progress bar: `role="progressbar"`, `aria-valuenow={pct}`, `aria-valuemin={0}`, `aria-valuemax={100}`, `aria-label="[filename] upload progress"`
- `PhotoEditModal` backdrop: `aria-modal="true"` on the panel, `role="dialog"`, `aria-labelledby` pointing to the modal heading id
- `PhotoEditModal` focus: on open, focus moves to the country `<select>` (Section A); on close, focus returns to the "Edit location" button that triggered it
- `PhotoEditModal` Escape: closes modal (same pattern as ConfirmDialog)
- Draggable pin in mini-map: supplemented with an instruction text `aria-describedby` — hidden `<span>` reading "Drag the pin to reposition the photo on the map" — `sr-only`
- "Edit location" button in PinPopup: `min-h-11` (44px) touch target, `aria-label="Edit location for [filename]"`
- Mobile drag handle: `aria-label="Drag to resize panel"`, `role="separator"` — non-interactive in Phase 4 (visual only), so add `aria-hidden="true"` instead since it has no function
