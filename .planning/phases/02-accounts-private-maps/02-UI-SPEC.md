# Phase 2 — UI Design Contract
## Accounts & Private Maps

> Visual and interaction contract for the Phase 2 auth screens and authenticated chrome.

## Design System

| Property | Value |
|----------|-------|
| Tool | **Tailwind CSS v4** (utility-first, CSS-first `@theme`) |
| Preset | Phase 1 tokens in `client/src/index.css` — **reused verbatim** (source: live codebase) |
| Component library | none — plain React components styled with Tailwind utility classes |
| Icon library | none — text labels only; Phase 4 may add Lucide |
| Font | system-ui stack (no web font download; matches Phase 1) |

**Design system status:** Tailwind v4 `@theme` tokens are live in `client/src/index.css`. All Phase 1 colors, type scale, and spacing are already defined. Phase 2 adds **no new tokens** — auth screens use the same palette and scale verbatim. No shadcn, no new registries.

**Font stack (body and UI):**
```css

```

## Spacing Scale

Identical to Phase 1. Tailwind's default 4px-base scale, custom semantic tokens unchanged.

| Token | Value | Tailwind utility (e.g. padding) | Usage in Phase 2 |
|-------|-------|---------------------------------|------------------|
| xs | 4px | `p-1` / `gap-1` | Icon gaps, checkbox label gap |
| sm | 8px | `p-2` / `gap-2` | Input inner padding, form field gap |
| md | 16px | `p-4` / `gap-4` | Card body padding, field spacing |
| lg | 24px | `p-6` / `gap-6` | Auth card padding, section header |
| xl | 32px | `p-8` / `gap-8` | Vertical spacing between card and edge |
| 2xl | 48px | `p-12` | Auth screen top/bottom breathing room |
| 3xl | 64px | `p-16` | Rarely used; auth screens are compact |

Exceptions:
- **Touch targets:** minimum 44px height for all interactive elements (buttons, checkbox, links). Use `min-h-11` (44px) Tailwind utility.
- **Auth card max-width:** `max-w-sm` (384px) — not on the 4-scale; a standard Tailwind width class.

Source: Phase 1 UI-SPEC.md (reused verbatim).

## Typography

Identical to Phase 1. Four roles, two weights. No new font sizes introduced.

| Role | Size | Weight | Line Height | Tailwind class | Usage in Phase 2 |
|------|------|--------|-------------|----------------|------------------|
| Body | 16px | 400 | 1.5 | `text-body font-normal` | Form instructions, link text, success copy |
| Label | 14px | 600 | 1.4 | `text-label font-semibold` | Button text, input labels, checkbox label, helper text |
| Heading | 20px | 600 | 1.2 | `text-heading font-semibold` | Auth card title ("Sign in", "Create account", etc.) |
| Display | 28px | 600 | 1.1 | `text-display font-semibold` | Not used in Phase 2 auth screens (no full-page headlines) |

**Rules (same as Phase 1):**
- Only `text-body`, `text-label`, `text-heading`, `text-display` — no other sizes.
- Only `font-normal` (400) and `font-semibold` (600) — no bold (700) or light (300).

## Layout

### Auth Screen Shell (unauthenticated gate — D-04)

Unauthenticated visitors see only the auth screens. The full app (map + photos) is hidden until login. The auth screens use a **centered card layout** on the app background — NOT the map. The map is not rendered at all in the logged-out state.

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│                    bg-bg (full viewport)                │
│                    height: 100dvh                       │
│                                                         │
│              ┌───────────────────────┐                  │
│              │   Auth Card           │                  │
│              │   max-w-sm (384px)    │                  │
│              │   bg-surface          │                  │
│              │   rounded-lg          │                  │
│              │   border-border       │                  │
│              │   shadow-sm           │                  │
│              │   p-6 (24px all sides)│                  │
│              └───────────────────────┘                  │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

- **Centering:** `flex items-center justify-center min-h-dvh` on the root auth wrapper.
- **Card:** `w-full max-w-sm bg-surface border border-border rounded-lg shadow-sm p-6`
- **Mobile:** Card is full-width with `mx-4` horizontal margin on screens narrower than `max-w-sm`. Same card; no separate mobile layout.
- **Background:** `bg-bg` (`#f8f9fa`) — the same neutral grey used in Phase 1. Subtle visual continuity; the user recognises it as the same app.

### Authenticated Chrome (logout affordance — overlay on the Phase 1 map)

Once logged in, the Phase 1 map renders as-is. A **minimal account control** is overlaid on the map — it must not compete with map readability.

```
┌─────────────────────────────────────────────────────────┐
│ ┌──────────────┐                                         │
│ │ Account strip│  position: fixed; top:8px; left:8px     │
│ │ z-index: 500 │  bg-surface, border, rounded, shadow    │
│ │ "Sign out"   │                                         │
│ └──────────────┘                                         │
│                                                         │
│                    WorldMap (full-bleed)                 │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

- **Position:** `fixed; top: 8px; left: 8px; z-index: 500` — top-left corner, above Leaflet tiles (z-index 400) and controls.
- **Appearance:** `bg-surface border border-border rounded-md shadow-sm px-3 py-2` — compact pill/chip.
- **Content:** user's email address (truncated to 20 chars with ellipsis if longer) at `text-label font-semibold text-text`, followed by a separator `·` and a "Sign out" button at `text-label text-text-muted hover:text-text`.
- **Width:** `max-w-[220px]` — keeps it compact and non-intrusive.
- **Interaction:** clicking "Sign out" calls logout → clears httpOnly cookie → returns to login screen.
- **Do NOT place on the right side** — the Phase 1 country panel occupies the right side at 360px; top-left avoids collision.

## Screen Specifications

### Screen 1: Login

**Route / trigger:** Default screen for all unauthenticated visitors.

**Card title:** "Sign in" (`text-heading font-semibold text-text`)

**Fields:**
| Field | Type | Label | Placeholder | Validation |
|-------|------|-------|-------------|------------|
| Email | `<input type="email">` | "Email" | "you@example.com" | Required; valid email format |
| Password | `<input type="password">` | "Password" | "••••••••" | Required; min 8 chars |

**Controls:**
- "Remember me" checkbox (D-05): `<input type="checkbox">` with label "Remember me" at `text-label`. Unchecked by default. Controls session TTL (~1 day vs ~30 days).
- Submit button: full-width, "Sign in" — see Button Spec below.
- Below button: text "Don't have an account?" + link "Create one" → navigates to Signup screen.
- Below "Create one": link "Forgot your password?" at `text-label text-text-muted` → navigates to Forgot Password screen.

**Layout (inside card, top to bottom):**
```
Card title: "Sign in"
─────────────
Email field (label + input)
Password field (label + input)
[Remember me checkbox + label]        [aligned left]
[Sign in button]                      [full-width]
Don't have an account? [Create one]   [centered, text-body]
[Forgot your password?]               [centered, text-label text-muted]
```

## States & Transitions

### Form Submitting State

When a form is submitted:
1. Submit button becomes disabled + text changes to submitting copy (see Copywriting).
2. All form inputs become `disabled` (prevent edits during flight).
3. No spinner animation required in Phase 2 (text label is sufficient).

### Form Error State

Field-level errors appear **below the affected input**, not in a top-of-card banner:
- `text-label text-destructive mt-1` — one line of copy per field.
- Input border changes to `border-destructive`.
- Errors appear after the first submit attempt; cleared on field re-edit.

Auth-level errors (wrong credentials, email already in use) appear as a **card-level error message** above the submit button:
- `text-body text-destructive` in a `rounded-md bg-red-50 border border-destructive px-4 py-3` block.
- Do NOT use a separate error component or toast for these — inline in the card.

### Success Toast (post-login / post-signup)

No toast for login/signup — the UI transition (auth screens → map) is itself the success signal. Toasts are reserved for password reset completion (see Copywriting).

| State | Copy |

## Accessibility Contract

- All form inputs: `<label for="...">` / `htmlFor` wiring — never unlabeled.
- Required fields: `required` attribute on `<input>` for native browser validation layer; custom inline errors are additive, not replacement.
- Error messages: `role="alert"` on card-level error block so screen readers announce on inject.
- Field-level errors: `aria-describedby` linking input to its error paragraph.
- Buttons: descriptive text labels — no icon-only buttons in Phase 2.
- Focus order: natural DOM order (card title → fields top-to-bottom → button → links).
- Color contrast: all text tokens from Phase 1 already pass WCAG AA (confirmed in Phase 1 UI-SPEC).
- "Remember me" checkbox: `<label>` wrapping ensures full-row click target.

## Responsive Breakpoints

Auth screens use a single centered-card layout. No layout change needed between mobile and desktop — the card is fluid (`w-full max-w-sm`) with `mx-4` gutters.

| Breakpoint | Behavior |
|------------|----------|
| All viewports | Single centered card; `mx-4` prevents card edge-to-edge on narrow screens |
| >= 768px (md) | Card max-width `max-w-sm` (384px) centered; no visual change vs mobile |

Authenticated chrome (account strip): always `fixed top-2 left-2` — works at all viewport widths since the Phase 1 country panel occupies the right side.
